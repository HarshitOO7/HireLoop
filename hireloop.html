Core Differentiator

Most tools either spam apply blindly or just generate resumes. HireLoop forces truth-checking of every skill before a resume is ever generated — with your own words as evidence. That evidence is stored in a persistent graph and reused automatically in every future application where the same skill appears.

Core Data Model
Persistent Skill Graph
Skills aren't a flat list. Each skill links to evidence, company, duration, and verification status. Evidence compounds — one confirmation reused in every future application requiring that skill.
Graph Node Structure
Schema SkillNode ├── skill_name: "Apache Kafka" ├── confidence: 0.85 // 0–1 scale ├── verified: true ├── source: "user_verified" └── evidence[ ] ├── company: "Acme Corp" ├── context: "async order pipeline" ├── duration_months: 8 └── last_used: "2024-06"
How Evidence Compounds
ParseInitial extraction from uploaded resume. Tagged resume_parsed, unverified.
VerifyYou confirm during skill dialog for a job. Your words saved as evidence permanently.
ReuseNext job needing Kafka — agent already has evidence, skips asking again, uses stored context directly in resume.
LearnInterview outcome → confidence score update. Rejected → no change (too many confounding variables).

Why a Graph Beats a Flat Skill List

Flat list problem: "Kafka: Yes" tells Claude nothing about depth. It might write "familiar with Kafka" whether you used it once or built a production pipeline with it for 8 months.

Graph advantage: Claude sees Kafka · 8mo · Acme · async order pipeline and writes "Designed Kafka-based asynchronous order processing pipeline at Acme Corp, delivering X% throughput improvement." That difference wins interviews.

Evidence reuse: Once verified once, Kafka evidence travels to every future job requiring it — no re-asking, stored context used automatically.

Truth enforcement: Go was deleted. The graph is the ground truth. Claude cannot accidentally claim it in any resume — it simply doesn't exist in the graph to include.

Architecture & UX
Improvement Notes
All changes from v1, plus additional actionable upgrades. Ordered by priority.

🔴 Baked Into v2 Design

Job Intake — Fixed
Filters Run Before Scraping — Not After
Job filter criteria (roles, location, salary floor, remote-only, blocked companies) defined once during onboarding. The intake node scrapes only jobs that pass all filters. SerpAPI quota isn't wasted, Claude never sees irrelevant jobs, and you never get pinged for noise.
Cover Letter — Fixed
Conditional Cover Letter — Only When JD Requires It
Claude parses the JD and flags requires_cover_letter: true/false. If true → generated automatically alongside resume, both delivered as PDFs. You can also set a global "always include one" preference. No manual decision per job, no missed requirements.
Skill Graph — Added
Rich Evidence Objects, Not Flat Strings
Each verified skill stores { company, role, context, duration_months, last_used, confidence, source }. This gives Claude precise data to write specific, credible bullets instead of vague claims. Better bullets → better interviews. Evidence is reused automatically in every future job requiring that skill.
Resume Variants — Added
Multiple Resume Bases with Auto-Selection
Upload 2–4 resume variants tagged by role type (backend, lead, fullstack, etc.). Claude reads the JD, checks your available variant tags, picks the best matching base, then tailors from there. Dramatically better quality than always starting from one generic resume.

🟡 Phase 2 — High Impact

Recruiter Discovery
Scan JD First, Web-Search as Fallback
Parse full JD for recruiter name, email, or LinkedIn. Most company job posts include a contact. If not found → Claude web searches [company] [role] recruiter site:linkedin.com. Results sent as a Telegram card with a direct outreach CTA. Increases response rate significantly compared to cold ATS applications.
Email Integration
Gmail Read-Only to Close the Outcome Loop
n8n Gmail node scans for email subjects containing applied company names. Classifies: interview invite → log interview, regret letter → log rejected. Feeds back to skill graph. Over time you see which resume versions convert — this is a compounding moat no other tool builds.
Rate Control
Built-In Application Rate Limiter
Hard limits: configurable daily application cap (default 5/day), no re-apply same company within 45 days. Managed by SQLite state in n8n. Without this, even manual applying patterns can get your email domain flagged by ATS systems or trigger LinkedIn rate limits.
Multi-User (Friends)
Telegram Supergroup Topics Before Building Multi-Tenant SaaS
Create a Telegram Supergroup with Topics (threads). Each friend gets their own topic thread. n8n routes all messages by message_thread_id. Handles 10–20 early users with zero infrastructure changes — no separate bots, no multi-tenant backend, no Kubernetes. Defer the complexity until there's real demand.

Core Tables

Jobs, Applications, Filters

SQL CREATE TABLE jobs ( id TEXT PRIMARY KEY, user_id TEXT, title TEXT, company TEXT, url TEXT, raw_jd TEXT, parsed JSON, -- Claude output fit_score REAL, requires_cl BOOLEAN, recruiter_name TEXT, -- Phase 2 recruiter_contact TEXT, -- Phase 2 status TEXT, -- pending|approved|skipped created_at DATETIME ); CREATE TABLE applications ( id INTEGER PRIMARY KEY, job_id TEXT, resume_used TEXT, -- file path cover_letter TEXT, -- file path or null applied_at DATETIME, outcome TEXT, -- interview|rejected|ghost email_signal TEXT -- Phase 2: raw snippet ); CREATE TABLE filters ( id INTEGER PRIMARY KEY, user_id TEXT, filter_type TEXT, -- role|location|salary|exclude filter_value TEXT, active BOOLEAN DEFAULT TRUE );

Claude — Fit Analysis Prompt

Prompt / System ## ROLE You are a job fit analyzer. Be conservative and honest. Never inflate scores. Return ONLY valid JSON. No preamble, no markdown fences. ## OUTPUT FORMAT { "fit_score": 0-100, "matched_skills": ["skill_name"], "missing_required": [{"skill":"...","importance":"required|preferred|nice"}], "requires_cover_letter": true/false, "best_resume_variant": "backend|lead|fullstack|...", "gap_summary": "2 sentence honest assessment", "action": "apply|consider|skip", "verify_questions": ["Have you used X in production? Where?"], "recruiter_info_found": "name/contact from JD, or null" } ## INPUTS Job Description: {{jd_text}} User Skill Graph: {{skill_graph_json}} Resume Variants Available: {{variant_tags}}

Claude — Resume Tailoring Prompt

Prompt / System ## ROLE Expert resume writer. Tailor truthfully using only verified evidence from the skill graph. ## HARD RULES - Only include skills where verified=true OR confidence >= 0.7 in graph - Never invent accomplishments — use STAR format for all bullets - Mirror JD keywords for ATS optimization (do not keyword-stuff) - Max 1 page under 5yr experience, 2 pages for 5yr+ - If cover letter required: append after ---COVER LETTER--- separator ## INPUTS Base Resume ({{variant_tag}}): {{base_resume_text}} Job Description: {{jd_text}} Verified Skill Graph: {{verified_skills_json}} User Evidence Notes: {{user_evidence_text}} Cover Letter Required: {{requires_cl}} ## OUTPUT Full tailored resume in Markdown. If CL required → append after ---COVER LETTER--- separator. Append ---CHANGES--- with 5 specific edits made and why.

Phase 3
Auto-Apply Architecture
Full automatic application on approved jobs. Human still approves each job individually — the agent handles all the mechanical work of filling forms and submitting. Never fires without your explicit per-job green light.
⚡
The Golden Rule of Auto-Apply

The agent never auto-applies to any job that hasn't been explicitly approved by you in Telegram. Approval happens per-job, after you've seen the tailored resume and chosen "✅ Apply Now". No batch approvals, no "apply to all". The automation only handles the mechanical form-filling — every trigger decision stays human.

Auto-Apply Pipeline (triggered after your approval)
1
You tap "✅ Apply Now" in Telegram

Telegram button sends approval signal to n8n. n8n logs approved_for_auto_apply = true + timestamp + resume version chosen. Daily application counter checked — if at limit, queued for tomorrow.

Gate
2
Platform Classifier

Agent reads the job URL and classifies the platform: Workday, Greenhouse, Lever, Ashby, BambooHR, direct company site, LinkedIn Easy Apply, or "Unknown". Each platform gets its own Playwright form-filling strategy. Unknown = fallback to manual with Telegram notification.

Claude
3
Form Recon

Playwright opens the application page. Claude reads the full form structure: all fields, dropdowns, required questions, resume upload slot, cover letter field, custom screening questions (e.g. "Why do you want to work here?"). Returns a field map.

Playwright
4
Answer Generation

Claude generates answers for all custom screening questions using your skill graph + profile. Standard fields (name, email, phone, years of experience) pulled from stored profile. Salary expectation: pulls from your preferences or flags for your input if missing. Any field Claude is unsure about gets sent to you for approval before submission.

Claude
5
Pre-Submit Review (Optional)

Optional setting: send a Telegram summary of all filled fields before clicking Submit. "Here's what I'm about to submit — confirm?" with a 5-minute timeout (auto-submits if no reply). Users can turn this off once they trust the agent.

Optional Gate
6
Submit + Screenshot

Playwright fills all fields, uploads resume PDF (and cover letter if required), submits the form. Takes a confirmation page screenshot. Sends screenshot to Telegram as proof of submission. Logs application to DB with timestamp + form field snapshot.

Playwright

Gmail integration watches for reply emails from the company domain. Subject-line classifier detects "interview", "next steps", "not moving forward". Sends Telegram notification immediately. Outcome logged to application record — feeds back into fit scoring model over time.

Safety Guardrails
Daily Cap

Hard limit of N applications/day (user-configurable, default 5). Exceeded = queued, not skipped. Prevents ATS blacklisting and LinkedIn flags.

Same-Company Cooldown

30-day block per company. Never applies to the same company twice in a month. Logged in DB.

Uncertain Field Escalation

Any form field Claude is <80% confident answering correctly gets sent to you before submission. No guessing on salary, visa status, or availability.

Screenshot Proof

Every submission gets a confirmation page screenshot sent to Telegram. You always know exactly what was submitted.

## What We're Building

HireLoop — a human-in-the-loop autonomous job hunting agent.

- Finds jobs → analyzes fit → verifies skills with YOU → generates tailored resume → you approve → (Phase 3) auto-applies
- Telegram is the PRIMARY interface — all triggers, approvals, uploads via Telegram bot
- Web dashboard comes in Phase 3 only — do NOT build it now
- No skill is ever claimed without explicit user confirmation
- No resume is sent without user approval
- No application fires without per-job approval

## Database Schema — Skill Graph

Build in db/models.py using SQLAlchemy async.

```python
class User(Base):
    __tablename__ = "users"
    id              = Column(String, primary_key=True)  # UUID
    telegram_id     = Column(String, unique=True)
    name            = Column(String)
    filters         = Column(JSON)     # role, location, salary, remote, blacklist
    notify_freq     = Column(String)   # "daily" | "realtime"
    min_fit_score   = Column(Integer, default=60)
    daily_app_limit = Column(Integer, default=5)
    onboarded       = Column(Boolean, default=False)
    created_at      = Column(DateTime)

class SkillNode(Base):
    __tablename__ = "skill_nodes"
    id           = Column(Integer, primary_key=True)
    user_id      = Column(String, ForeignKey("users.id"))
    skill_name   = Column(String)
    # verified_resume | verified_attested | partial | gap
    status       = Column(String)
    # high | medium | low — from resume parse
    confidence   = Column(String)
    created_at   = Column(DateTime)
    updated_at   = Column(DateTime)

class SkillEvidence(Base):
    __tablename__ = "skill_evidence"
    id               = Column(Integer, primary_key=True)
    skill_node_id    = Column(Integer, ForeignKey("skill_nodes.id"))
    company          = Column(String)
    role_title       = Column(String)
    duration_months  = Column(Integer)
    last_used_year   = Column(Integer)
    user_context     = Column(Text)   # user's own words
    generated_bullet = Column(Text)   # Claude's polished bullet
    source           = Column(String) # "resume" | "telegram" | "manual"

class Job(Base):
    __tablename__ = "jobs"
    id                    = Column(String, primary_key=True)  # UUID
    user_id               = Column(String, ForeignKey("users.id"))
    title                 = Column(String)
    company               = Column(String)
    url                   = Column(String)
    raw_jd                = Column(Text)
    parsed                = Column(JSON)    # output of parse_job()
    fit_score             = Column(Float)
    cover_letter_required = Column(Boolean, default=False)
    recruiter_name        = Column(String)  # Phase 2
    recruiter_linkedin    = Column(String)  # Phase 2
    # pending|skill_verify|approved|skipped|applied
    status                = Column(String)
    created_at            = Column(DateTime)

class Application(Base):
    __tablename__ = "applications"
    id                 = Column(Integer, primary_key=True)
    job_id             = Column(String, ForeignKey("jobs.id"))
    resume_path        = Column(String)
    cover_letter_path  = Column(String)
    applied_at         = Column(DateTime)
    # interview | rejected | ghosted | offer
    outcome            = Column(String)
    outcome_source     = Column(String)  # email | manual | telegram
    outcome_at         = Column(DateTime)
```

## Onboarding Flow (6 ConversationHandler states)

```
STATE: WELCOME
  Send welcome message + [✅ Let's go] button

STATE: UPLOAD_RESUME
  "Send 1–4 resume PDFs or Word docs"
  On each document: download → extract text → call ai.parse_resume()
  Button: [✅ Done uploading]

STATE: CONFIRM_SKILLS
  Show extracted skills grouped by confidence
  High confidence: auto-confirmed
  Medium/Low: show [✅ Confirm] [✏️ Add context] [❌ Remove] per skill
  If "Add context": ask for one sentence → save as user_context
  Save all confirmed to SkillNode + SkillEvidence

STATE: SET_FILTERS
  Ask role, location, remote pref, min salary, blacklist companies
  Save to user.filters JSON

STATE: SET_FREQUENCY
  [📬 Daily digest] [⚡ Real-time] [2x per day]
  Min fit score: [50%] [60%] [70%] [80%]

STATE: DONE
  "All set! Running first job search now..."
  Trigger first scrape
```

## Cover Letter Logic — CRITICAL

ONLY generate a cover letter when one of these is true:
1. job.cover_letter_required = True (JD mentioned it)
2. User explicitly tapped [📝 Add Cover Letter] button

NEVER auto-generate without one of those triggers.

Detection in parse_job() scans for:
"cover letter", "covering letter", "letter of motivation", "please include"

## Job Scraping

### Primary — JobSpy (free, use this)
```python
from jobspy import scrape_jobs

jobs = scrape_jobs(
    site_name=["indeed", "linkedin", "glassdoor", "google"],
    search_term=user_filters["role"],
    location=user_filters["location"],
    is_remote=user_filters.get("remote_only", False),
    results_wanted=25,
    hours_old=24,
)
```

### User-pasted URLs — Jina Reader
```python
import httpx
async def fetch_job_from_url(url: str) -> str:
    async with httpx.AsyncClient() as client:
        r = await client.get(f"https://r.jina.ai/{url}", timeout=30)
        return r.text
```

Filters run BEFORE passing jobs to fit analysis:
- Salary minimum
- Blacklisted companies/industries
- Seniority match
- Deduplicate by URL
- Fit score threshold (skip notification if below user.min_fit_score)

## Coding Rules — Always Follow These

1. Always use async/await — bot and AI calls are all async
2. Never hardcode a provider — always AIFactory.create() or inject AIProvider
3. Never include a skill in a resume unless SkillNode.status is "verified_*"
4. Never generate a cover letter unless job.cover_letter_required=True OR user requested
5. SQLAlchemy models only in db/models.py — no raw SQL strings
6. All InlineKeyboardMarkup in bot/keyboards.py — not inline in handlers
7. ConversationHandler for all multi-step Telegram flows
8. All secrets from .env only — never hardcode
9. Log everything to DB — every job seen, skill verified, application made
10. Filters run BEFORE scraping — pass to JobSpy, don't post-filter a huge list

# HireLoop — Claude Code Project Context > This file is read automatically by Claude Code. It contains the full architecture, > decisions, schema, and build order. SQLite for Phase 1 (DATABASE_URL=sqlite:///hireloop.db). class User(Base): id = Column(String, primary_key=True) # UUID telegram_id = Column(String, unique=True) name = Column(String) filters = Column(JSON) # {role, location, salary_min, remote, blacklist[]} notify_freq = Column(String) # "daily" | "realtime" | "twice_daily" min_fit_score = Column(Integer, default=60) daily_app_limit = Column(Integer, default=5) onboarded = Column(Boolean, default=False) created_at = Column(DateTime, default=datetime.utcnow) class SkillNode(Base): # GRAPH NODE — one row per unique skill per user id = Column(Integer, primary_key=True) user_id = Column(String, ForeignKey("users.id")) skill_name = Column(String) status = Column(String) # verified_resume|verified_attested|partial|gap confidence = Column(String) # high|medium|low (from parse_resume) created_at = Column(DateTime) updated_at = Column(DateTime) class SkillEvidence(Base): # GRAPH EDGE — proof that the skill was used id = Column(Integer, primary_key=True) skill_node_id = Column(Integer, ForeignKey("skill_nodes.id")) company = Column(String) role_title = Column(String) duration_months = Column(Integer) last_used_year = Column(Integer) user_context = Column(Text) # user's own words generated_bullet = Column(Text) # Claude's polished bullet (generated lazily) source = Column(String) # "resume" | "telegram" | "manual" class Job(Base): id = Column(String, primary_key=True) # UUID user_id = Column(String, ForeignKey("users.id")) title = Column(String) company = Column(String) url = Column(String) raw_jd = Column(Text) parsed = Column(JSON) # output of parse_job() fit_score = Column(Float) cover_letter_required = Column(Boolean, default=False) recruiter_name = Column(String) # Phase 2 recruiter_linkedin = Column(String) # Phase 2 status = Column(String) # pending|skill_verify|approved|skipped|applied created_at = Column(DateTime) class Application(Base): id = Column(Integer, primary_key=True) job_id = Column(String, ForeignKey("jobs.id")) resume_path = Column(String) cover_letter_path = Column(String) applied_at = Column(DateTime) outcome = Column(String) # interview|rejected|ghosted|offer outcome_source = Column(String) # email|manual|telegram outcome_at = Column(DateTime) --- ## AI Provider System (ai/) # AIProvider is abstract — subclass it for each provider class AIProvider(ABC): @abstractmethod async def complete(self, messages, system=None, json_mode=False) -> AIResponse: ... @property @abstractmethod def provider_name(self) -> str: ... # AIFactory reads .env and returns correct provider # AI_PROVIDER options: anthropic|openai|gemini|groq|ollama|together provider = AIFactory.create() ai = HireLoopAI(provider) # HireLoopAI wraps all 6 tasks job = await ai.parse_job(raw_jd_text) skills = await ai.parse_resume(resume_text) fit = await ai.analyze_fit(job, user_profile_dict) resume = await ai.tailor_resume(job, fit, base_resume, verified_skills) cl = await ai.write_cover_letter(job, user_profile, fit) answers = await ai.answer_screening_questions(questions, job, profile) # Default models: anthropic→claude-sonnet-4-6, openai→gpt-4o, # gemini→gemini-2.0-flash, groq→llama-3.3-70b-versatile, ollama→llama3.2 # Groq/Ollama/Together are subclasses of OpenAIProvider (same HTTP spec) --- ## Telegram Bot Commands + Keyboards # Commands (register with BotFather) /start - onboarding wizard /skills - view skill graph summary /resume - upload new resume anytime /jobs - pending jobs /history - past applications + outcomes /settings - edit all preferences /filters - quick filter edit /pause - pause job hunting /help - command reference # Persistent reply keyboard (always visible at bottom) MAIN_KEYBOARD = [ ["📎 Add Resume", "🎛️ Edit Filters"], ["📊 My Skills", "📋 Pending Jobs"], ["⏸ Pause Agent", "⚙️ Settings"], ] --- ## Job Scraping Strategy # PRIMARY: JobSpy (free, multi-source — use this first) # pip install jobspy jobs = scrape_jobs( site_name=["indeed", "linkedin", "glassdoor", "google"], search_term=user_filters["role"], location=user_filters["location"], is_remote=user_filters.get("remote_only", False), results_wanted=25, hours_old=24, ) # SECONDARY: Jina Reader for user-pasted URLs # No API key needed, no scraping bans, works on LinkedIn/Indeed jina_url = f"https://r.jina.ai/{user_pasted_url}" clean_text = httpx.get(jina_url).text # PHASE 2+: RapidAPI Jobs API, Apify actors, Bright Data --- ## Fit Scoring Rules - Required skill gap = -15 to -20 points - Nice-to-have gap = -3 to -5 points - Unconfirmed skills = ask user, then recalculate - Threshold default = 60% (user-configurable) - Below threshold = skip silently, no notification --- ## Skill Graph — How Evidence Travels # When user confirms: "Kafka at Acme, 8 months, order pipeline" # Creates: SkillNode(skill="Kafka", status="verified_attested") # + SkillEvidence(company="Acme", months=8, context="order pipeline") # Next time any job needs Kafka → evidence auto-populates resume bullet # User never explains Kafka again --- ## Coding Rules — Always Follow These 1. Always async/await — Telegram bot + AI calls are all async 2. Never hardcode a provider — always AIFactory.create() or inject AIProvider 3. Never claim a skill in a resume unless SkillNode.status = "verified_*" 4. Never send cover letter unless job.cover_letter_required=True OR user explicitly requested 5. SQLAlchemy models in db/models.py only — no raw SQL strings anywhere 6. All Telegram keyboards in bot/keyboards.py — never inline in handlers 7. ConversationHandler for multi-step flows (onboarding, skill verify) 8. Secrets only from .env via python-dotenv — never hardcode 9. Log everything to DB — every job, verification, application 10. Filters pass INTO JobSpy — do not post-filter a large result set --- ## Phase 1 Build Order Week 1: requirements.txt · .env.example · db/models.py · ai/ layer · scripts/test_provider.py Week 2: Telegram bot setup · bot/onboarding.py (6 states) · resume_upload handler · skill confirm Week 3: jobs/scraper.py (JobSpy) · jobs/parser.py (Jina) · fit analysis · skill verify dialog Week 4: resume/generator.py · pdf_export.py · job_approval handler · docker-compose.yml --- ## DO NOT build in Phase 1 ❌ Web dashboard (Phase 3) ❌ Auto-apply/Playwright (Phase 3) ❌ Recruiter finder (Phase 2) ❌ Gmail integration (Phase 3) ❌ Multi-user SaaS auth (P2+) --- ## Environment Variables (.env) AI_PROVIDER=anthropic # anthropic|openai|gemini|groq|ollama|together AI_API_KEY=sk-ant-...
